## 0. 개념 이해하기 — 우리가 왜 이 방법을 쓰는지

지금 팩인백은 Next.js로 만든 **웹사이트**이고, Vercel에 배포되어 있어요. 앱스토어는 "웹사이트 주소"를 올릴 수 없고, 반드시 **iOS 전용 앱 파일(.ipa)**을 올려야 해요.

그래서 **Capacitor**라는 도구를 씁니다. Capacitor는 우리 웹앱을 아이폰 안의 "전용 웹 브라우저 창(WKWebView)"에 통째로 넣어서, 겉보기에는 진짜 네이티브 앱처럼 보이게 만들어주는 도구예요.

```
[App Store에서 다운로드한 앱]
        │
        ▼
   Capacitor 껍데기 (Xcode 26로 빌드한 진짜 iOS 앱)
        │  이 안에서 우리 웹사이트 주소를 그대로 불러옴 (server.url 방식)
        ▼
   https://packinbag.vercel.app  ← 지금 쓰고 있는 그 웹사이트
        │
        ▼
   Firebase (로그인/데이터/이미지) ← 지금과 완전히 동일
```

**중요한 장점:** `server.url` 방식(우리가 쓰는 방식)을 쓰면, 나중에 기능을 추가하거나 버그를 고칠 때 **Vercel에만 새로 배포하면 앱스토어 재심사 없이 앱 내용도 바로 업데이트**돼요. Xcode 재빌드가 필요한 건 아이콘 변경, 네이티브 권한(카메라 등) 추가, iOS 자체 설정 변경 같은 경우뿐이에요.

**알아둬야 할 단점(애플 심사 이슈):** 애플은 "그냥 웹사이트를 감싸기만 한 앱"을 좋아하지 않아요 (가이드라인 4.2, "최소한의 기능성"). 100% 거절되는 건 아니지만, 통과 확률을 높이려면:
- 스플래시 화면, 상태바 색상 등 네이티브스러운 디테일을 챙기고 (기본 설정되어 있음)
- 로그인 없이도 앱의 핵심 가치를 볼 수 있게 하거나, 리뷰어용 데모 계정을 제공하고
- 가능하면 나중에 푸시 알림 등 네이티브 기능을 1~2개 추가하는 게 좋아요

## 0.5. ⚠️ 시작 전에 반드시 처리해야 할 것 — 구글 로그인 문제

**이건 다른 모든 단계보다 먼저 해결해야 하는 문제예요.** 지금 팩인백의 구글 로그인은 Firebase JS SDK의 `signInWithPopup()` 방식만 구현돼 있어요. 이건 **웹(Safari, Chrome 등)에서는 정상 작동**하지만, Capacitor로 감싼 **iOS 네이티브 앱(WKWebView) 안에서는 구글이 정책적으로 차단**해요.

### 왜 막히나요?

구글은 몇 년 전부터 "앱 안에 내장된 웹뷰(embedded webview)"에서 구글 계정으로 로그인하는 것을 보안 정책상 금지하고 있어요. 앱 개발자가 웹뷰 안의 내용을 마음대로 조작할 수 있어서, 사용자가 구글 아이디/비밀번호를 입력하는 화면을 가로챌 위험이 있기 때문이에요. WKWebView(Capacitor가 쓰는 것)는 정확히 이 "금지된 웹뷰" 범주에 들어가요.

실제로 시도하면 아래 두 에러 중 하나가 나요:
- `이 브라우저 또는 앱은 안전하지 않을 수 있습니다`
- `disallowed_useragent` (개발자 콘솔에서 보이는 에러 코드)

**이건 팩인백 코드가 잘못돼서가 아니라, 구글이 전세계 모든 앱에 똑같이 적용하는 정책이에요.** 애플 심사와는 별개로, **Capacitor 앱으로 빌드하는 순간부터 구글 로그인이 아예 작동하지 않아요.** 즉 지금 상태로 TestFlight에 올려도 로그인 화면에서 막혀서 그 이후 화면을 아예 테스트할 수 없어요.

### 해결 방법

WKWebView 안에서 직접 구글 로그인 창을 띄우는 대신, **아이폰의 진짜 브라우저(Safari 엔진, `ASWebAuthenticationSession`)를 잠깐 띄워서 로그인시키고, 끝나면 다시 앱으로 돌아오는 방식**으로 바꿔야 해요. 직접 그 로직을 짜는 대신, 이미 만들어진 Capacitor 플러그인을 쓰는 걸 추천해요. 대표적으로:

- **`@capgo/capacitor-social-login`** (요즘 가장 많이 쓰이고 활발히 관리되는 플러그인, Google/Apple/Facebook 등 지원)
- **`@codetrix-studio/capacitor-google-auth`** (오래되고 검증된 대안)

둘 다 내부적으로 iOS의 `GoogleSignIn` SDK(또는 `ASWebAuthenticationSession`)를 써서, 구글이 허용하는 방식으로 로그인 창을 띄우고, 성공하면 Firebase에 그대로 연결할 수 있는 idToken을 돌려줘요. 대략적인 흐름은:

```
1. 웹(플러그인 미설치/브라우저)에서는 → 기존 signInWithPopup 그대로 사용
2. 네이티브 앱(Capacitor.isNativePlatform() === true)에서는
   → 위 플러그인으로 구글 로그인 → idToken 받기
   → firebase.auth.GoogleAuthProvider.credential(idToken)로 Firebase 로그인 완료
```

이 작업은 **코드 작업(플랫폼 분기 + 플러그인 연동)이 필요한 부분이라, 다음 대화에서 저(Claude)와 같이 진행하는 걸 추천해요.** 애플 심사 제출 전에 반드시 끝내야 하는 항목이에요. (Apple/Kakao/Naver 로그인 확장 계획이 있으셨는데, 이 작업을 하는 김에 Apple 로그인도 같이 붙이면 애플 가이드라인 4.8 "제3자 로그인을 제공하면 Apple 로그인도 제공해야 함" 요건까지 한 번에 해결돼요 — 아래 12번 FAQ 참고)

> 💡 급하게 먼저 테스트만 해보고 싶다면: 임시로 "이메일/비밀번호 로그인" 계정을 하나 만들어서 그 계정으로 TestFlight 테스트를 진행하고, 구글 로그인 수정은 별도로 진행해도 됩니다. 하지만 **실제 사용자 대부분이 구글 로그인을 쓸 거라, 정식 출시 전에는 반드시 고쳐야 해요.**

> 💡 회사 컴퓨터(Windows)는 코드 작성용으로 계속 써도 되고, **Xcode를 열어서 빌드/제출하는 마지막 단계만 Mac에서** 하면 돼요.

### Xcode 26 설치 방법 (둘 중 하나)

**방법 A — Mac App Store (가장 쉬움)**
1. Mac App Store 앱 열기 → "Xcode" 검색 → 설치
2. 이미 Xcode가 있다면 App Store의 "업데이트" 탭에서 최신 버전(26.x)으로 업데이트

**방법 B — Apple 개발자 사이트에서 `.xip` 직접 다운로드 (더 빠름, 여러 버전 보관 가능)**
1. [developer.apple.com/download](https://developer.apple.com/download/all/) 접속 (Apple Developer 계정 로그인 필요)
2. "Xcode 26" 항목의 `.xip` 파일 다운로드
3. 다운로드 완료 후 더블클릭하면 자동으로 압축 해제됨 (몇 분 소요)
4. 압축 해제된 `Xcode.app`을 `/Applications` 폴더로 이동
5. 터미널에서 커맨드라인 도구 경로를 지정:
   ```bash
   sudo xcode-select -switch /Applications/Xcode.app
   ```

설치 후 터미널에서 버전 확인:
```bash
xcodebuild -version
# Xcode 26.x
# Build version ...
```

## 2. Apple Developer Program 가입

1. [developer.apple.com/programs](https://developer.apple.com/programs) 접속
2. **"Enroll"(등록)** 클릭 → 평소 쓰는 Apple ID로 로그인 (없으면 새로 생성)
3. 가입 유형 선택:
   - **개인(Individual)**: 본인 이름으로 앱이 출시됨. 서류 절차가 간단, 보통 하루 안에 승인
   - **회사(Organization)**: 회사/팀 이름으로 출시. D-U-N-S 번호라는 사업자 확인 서류가 필요해서 며칠~몇 주 걸릴 수 있음
   - → 처음이고 개인 프로젝트라면 **개인(Individual)** 추천
4. 연락처/주소 등 정보 입력 (신용카드 결제이므로 영문 주소 입력 권장)
5. 결제 진행: **연 99달러** (한국 카드로 결제 시 원화 약 14만원 내외, 환율에 따라 변동)
6. 승인 완료 이메일이 오면 가입 끝 (보통 몇 시간~1일 소요, 서류 확인이 필요하면 더 걸릴 수 있음)

✅ 완료 조건: [developer.apple.com/account](https://developer.apple.com/account) 접속 시 멤버십이 "Active"로 표시됨

> 참고: appfle.dev 구글 계정이 정지되면서 여러 서비스를 appfle.io로 이전하셨죠. Apple Developer 계정은 이메일 주소와 무관하게 **Apple ID** 기준이라, 구글 계정 정지와는 완전히 별개예요. 다만 Apple ID의 이메일이 appfle.dev 쪽으로 걸려있다면, 나중에 로그인이 안 될 위험을 피하기 위해 Apple ID 설정에서 연락처 이메일을 appfle.io 쪽으로 미리 바꿔두는 걸 추천해요.

## 3. 프로젝트에 Capacitor 설치하기

프로젝트에 Capacitor 관련 설정 파일들이 이미 들어있어요 (`capacitor.config.ts`, `package.json`). 터미널에서 아래 명령어로 실제 패키지를 설치하세요.

```bash
cd packinbag
npm install
```

> 참고: 지금 `package.json`에는 Capacitor **6.x** 버전이 명시돼 있어요. Xcode 26/iOS 26 SDK로 빌드하는 것 자체는 Capacitor 6에서도 문제없이 잘 됩니다 (Capacitor 개발팀이 확인한 내용 — Capacitor 코드 자체의 마이그레이션은 필요 없고, **Mac에 설치된 Xcode 버전만 26 이상이면 됨**). 다만 만약 Xcode 26에서 이상한 빌드 에러가 난다면, Capacitor를 최신(7.x 또는 8.x)으로 올리는 것도 옵션이에요 — 이 부분은 실제로 에러가 나면 그때 저와 같이 봐도 충분해요.

설치가 끝나면 `capacitor.config.ts` 파일을 열어서 **딱 2곳**을 확인/수정하세요.

```ts
appId: "com.appfle.packinbag",   // 이미 설정되어 있음, 변경하지 마세요
...
server: {
  url: "https://packinbag.vercel.app", // 실제 배포된 Vercel 주소와 일치하는지 확인
}
```

> ⚠️ `appId`(Bundle Identifier)는 **앱스토어에 한번 등록하면 절대 변경할 수 없는 값**이에요. 이미 `com.appfle.packinbag`로 정해져 있으니 이후 단계에서 계속 이 값을 그대로 씁니다.

## 4. Xcode용 iOS 프로젝트 만들기

여기서부터는 **Mac에서** 진행하세요.

1. CocoaPods가 설치되어 있는지 확인 (Capacitor 6는 iOS 네이티브 의존성 관리를 CocoaPods로 함):
   ```bash
   pod --version
   ```
   안 나오면 설치:
   ```bash
   sudo gem install cocoapods
   ```
   (Mac 최신 버전 + Xcode 26 조합에서 `sudo gem install`이 권한 에러를 내면 `brew install cocoapods`로 Homebrew를 통해 설치하는 걸 추천해요)

2. 터미널에서 프로젝트 폴더로 이동 후:
   ```bash
   npx cap add ios
   ```
   이 명령어를 실행하면 프로젝트 안에 `ios/` 폴더가 새로 생겨요. 이 폴더가 바로 **Xcode가 인식하는 진짜 iOS 앱 프로젝트**예요.

3. 아래 명령어로 Xcode를 자동으로 열어줍니다:
   ```bash
   npx cap open ios
   ```
   Xcode 26이 열리면서 `App.xcworkspace` 라는 파일이 로드돼요. (⚠️ `App.xcodeproj`가 아니라 **`.xcworkspace`**로 열려야 정상이에요 — CocoaPods 의존성이 워크스페이스 단위로 묶여있기 때문)

4. Xcode 26의 첫 화면(웰컴 스크린)이 따로 뜬다면 닫고, 왼쪽에 파일 목록(Navigator), 가운데에 코드/설정 화면이 보이면 정상입니다. Xcode 26은 macOS 26(Tahoe)의 "Liquid Glass" 디자인이 적용돼서 창 테두리나 툴바 아이콘이 예전보다 더 투명하고 둥글게 보이는데, 기능적으로는 동일해요.

## 5. 앱 아이콘 만들기

앱스토어는 다양한 크기의 아이콘(홈화면용, 설정용, 알림용 등)을 요구하는데, **1024x1024 이미지 1장만 있으면 자동으로 다 만들어줘요.**

1. 정사각형(1:1) 이미지 1장을 준비하세요 (배경이 꽉 찬 디자인 권장, 텍스트는 최소화). 팩인백 브랜드 컬러(`#FF6E2D`)와 배낭 실루엣을 활용하면 좋아요.
2. 프로젝트 루트에 `resources` 폴더를 만들고 그 안에 `icon.png` (1024x1024)로 저장
3. 터미널에서:
   ```bash
   npm install @capacitor/assets --save-dev
   npx capacitor-assets generate --ios
   ```
4. 자동으로 `ios/App/App/Assets.xcassets` 안에 필요한 모든 사이즈가 채워집니다. (iOS 18 이후로 아이콘에 알파채널이 있으면 자동으로 "다크/투명 아이콘" 변형까지 요구하는데, `capacitor-assets`가 알아서 처리해줘요)

> 아이콘 디자인이 아직 없다면, 우선 팩인백 로고 느낌으로 심플한 배경색 + 배낭 아이콘 정도로 임시 제작 후 나중에 교체해도 됩니다. 필요하면 다음 대화에서 아이콘 디자인 시안도 같이 만들어드릴게요.

## 6. Xcode 26에서 서명(Signing) 설정

"서명"은 이 앱이 진짜 애플에게 승인받은 개발자가 만든 게 맞다는 걸 증명하는 절차예요. Xcode가 대부분 자동으로 처리해줘요.

1. Xcode 왼쪽 파일 목록(Navigator)에서 최상단 **"App"** 프로젝트 클릭
2. 가운데 화면 상단 탭에서 **"Signing & Capabilities"** 선택 (Xcode 26에서도 이름/위치 동일)
3. **"Automatically manage signing"** 체크박스 켜기
4. **Team** 드롭다운에서 본인 Apple Developer 계정 선택 (2번 단계에서 가입한 계정). 계정이 안 보이면 Xcode 메뉴 > Settings(⌘,) > Accounts에서 Apple ID를 먼저 추가하세요.
5. **Bundle Identifier**가 `com.appfle.packinbag`로 정확히 들어가 있는지 확인
6. 빨간 에러가 없어지면 완료 (에러가 있다면 대부분 "Bundle Identifier가 이미 사용 중" 문제 → appId를 더 고유하게 변경해야 하는데, 이미 정해진 값이니 신중하게)
7. 왼쪽 Navigator에서 **"Info"** 탭도 확인: `CFBundleDisplayName`(앱 이름 표시용)이 "팩인백"으로 되어있는지, `NSAppTransportSecurity` 관련 설정이 있다면 HTTPS(Vercel)만 쓰므로 특별히 손댈 필요 없어요.

### Xcode 26에서 새로 신경써야 할 점: 최소 배포 버전(Deployment Target)

Xcode 26은 기본적으로 최신 iOS SDK를 기준으로 프로젝트를 만들지만, **최소 지원 iOS 버전은 직접 낮출 수 있어요.** 너무 최신 버전만 지원하면 구형 아이폰 사용자가 설치를 못 해요.
1. "App" 프로젝트 → **"Build Settings"** 탭 → 검색창에 "Deployment Target" 입력
2. **iOS Deployment Target**을 원하는 최소 버전으로 설정 (특별한 이유가 없다면 **iOS 16.0** 정도가 최신 기능과 구형 기기 지원의 합리적인 균형점이에요)
3. `ios/App/Podfile` 파일도 열어서 상단의 `platform :ios, '...'` 값을 동일하게 맞춰주세요

## 7. 내 아이폰/시뮬레이터로 테스트

1. Xcode 상단 중앙의 기기 선택 드롭다운에서:
   - **시뮬레이터**로 테스트하려면 "iPhone 17" 또는 "iPhone 17 Pro" 같은 시뮬레이터 선택 (Xcode 26에는 iPhone 17 라인업 시뮬레이터가 기본 포함됨)
   - **실제 내 아이폰**으로 테스트하려면 Mac에 케이블(또는 같은 Wi-Fi)로 연결 후 기기 이름 선택 (아이폰에서 "이 컴퓨터를 신뢰하시겠습니까?" 뜨면 신뢰 클릭)
2. 좌측 상단 ▶️ (재생 버튼) 클릭
3. 빌드가 끝나면 앱이 자동 실행되면서 팩인백 로그인 화면이 뜨면 성공!

실기기 테스트 시 처음엔 "신뢰할 수 없는 개발자입니다" 에러가 뜰 수 있어요 → 아이폰의 **설정 > 일반 > VPN 및 기기 관리**에서 본인 개발자 계정을 신뢰 처리하면 해결돼요.

> ⚠️ 이 단계에서 구글 로그인을 테스트해보면 **0.5번에서 설명한 문제가 바로 재현될 거예요.** 코드 수정이 안 된 상태라면 이메일/비밀번호 계정으로 우선 테스트하세요.

**키보드 관련 최근 수정사항 확인**: 최근 팩 편집 화면의 키보드 대응 로직(visualViewport + 강제 스크롤 취소)을 손봤어요. 실기기에서 짐 추가 입력창에 포커스를 줬을 때 헤더가 밀리지 않고 입력창이 키보드 바로 위에 잘 붙어있는지 꼭 확인해주세요. 문제가 있으면 알려주시면 같이 더 다듬을게요.

## 8. Mac 버전 켜기 (Mac Catalyst)

아이폰/아이패드용 프로젝트에 체크박스 하나만 추가하면 **동일한 코드로 Mac 앱까지 자동 지원**돼요 (이 기술을 Mac Catalyst라고 불러요).

1. Xcode에서 "App" 프로젝트 → **"General"** 탭
2. **"Supported Destinations"** 항목에서 **"+"** 버튼 클릭
3. **"Mac (Designed for iPad)"** 추가
4. 상단 기기 선택 드롭다운에서 "My Mac"을 선택하고 ▶️ 실행해서 Mac에서 잘 뜨는지 확인

> Mac Catalyst 버전은 아이패드 UI를 그대로 재사용하는 방식이라 추가 코드 작업이 거의 필요 없어요. 다만 마우스/트랙패드/키보드 환경에서 레이아웃이 어색한 부분이 있으면 나중에 다듬으면 됩니다.

> 참고: "Mac (Designed for iPad)"를 추가하면 사실상 앱이 **iPad도 지원하는 상태**가 돼요. 이 경우 9번 단계에서 **iPad 스크린샷도 필수로 제출**해야 한다는 점을 기억해두세요.

## 9. App Store Connect에 앱 등록

1. [appstoreconnect.apple.com](https://appstoreconnect.apple.com) 접속 → Apple Developer 계정으로 로그인
2. **"나의 앱"(My Apps)** → **"+"** → **"신규 앱"**
3. 정보 입력:
   - **플랫폼**: iOS (Mac Catalyst는 자동 포함됨)
   - **이름**: 팩인백 (앱스토어에 노출될 이름, 나중에 변경 가능)
   - **기본 언어**: 한국어
   - **Bundle ID**: `com.appfle.packinbag` (Xcode에서 설정한 것과 동일한 항목을 드롭다운에서 선택)
   - **SKU**: 아무 고유 문자열 (예: `packinbag001`, 사용자에게 안 보이는 내부 관리용 값)
4. **"만들기"** 클릭

이후 앱 상세 페이지에서 아래 항목들을 채워야 심사 제출이 가능해요:

### 스크린샷 (2026년 기준 최신 규격)

애플이 최근에 규격을 단순화해서, **가장 큰 사이즈 1세트만 올리면 나머지는 자동으로 축소**돼요.

| 기기 | 필수 여부 | 픽셀 크기 (세로/portrait) |
|---|---|---|
| **iPhone 6.9인치** (iPhone 17 Pro Max 등) | 이 크기만 있으면 나머지 iPhone은 자동 스케일링됨 | **1320 x 2868** |
| iPad 13인치 (iPad Pro M4/M5) | **iPad를 지원하면(8번에서 Mac Catalyst 켰다면) 필수** | **2064 x 2752** |

- 스크린샷은 시뮬레이터(⌘S로 캡처) 또는 실기기에서 캡처하면 됩니다. 시뮬레이터 기기 종류를 "iPhone 17 Pro Max"로 바꿔서 캡처하면 정확한 크기가 나와요.
- 형식은 PNG 또는 JPEG, 언어별로 최소 1장~최대 10장.
- 픽셀이 단 1px이라도 틀리면 업로드가 거부되니, 캡처 후 크기를 재조정(리사이즈)하지 말고 그대로 올리세요.

### 그 외 필수 항목

- **앱 설명, 키워드, 카테고리** (예: 카테고리 = "생산성" 또는 "여행")
- **개인정보처리방침 URL**: `/privacy` 페이지 주소를 입력하면 됩니다.
  예: `https://packinbag.vercel.app/privacy`
  (내용은 appfle님 실제 연락처로 꼭 최신 상태인지 확인해주세요)
- **앱 프라이버시(Privacy Nutrition Label)**: "앱 프라이버시" 섹션에서 팩인백이 수집하는 데이터 종류를 체크하는 설문이에요. 팩인백 기준으로는 대략:
  - 이메일 주소, 이름/닉네임(계정 기능) → "본인 확인" 또는 "앱 기능" 목적으로 수집한다고 표시
  - 사진(가방 이미지 업로드) → "사용자 콘텐츠"로 표시
  - Firebase Analytics를 아직 안 붙였다면 "분석" 항목은 체크 안 해도 됨
  이 설문에 정직하게 답하지 않으면(예: 실제로 수집하는데 "수집 안 함"으로 표시) 나중에 앱이 강제로 내려갈 수 있으니 신중하게 채워주세요.
- **연령 등급 설문**: 2026년부터 등급 체계가 더 세분화됐어요(기존 4+/9+/12+/17+ 대신 더 세밀한 카테고리). 특별한 내용(폭력, 도박 등) 없으면 대부분 자동으로 가장 낮은 등급으로 설정됨.
- **Apple 로그인 관련 체크**: 구글 로그인 같은 제3자 로그인을 제공하면서 Apple 로그인을 제공하지 않으면 가이드라인 4.8 위반으로 거절될 수 있어요. 0.5번 작업(구글 로그인 네이티브화) 하는 김에 Apple 로그인도 검토해보세요.

## 10. 빌드 업로드 + TestFlight 테스트

1. Xcode 상단 기기 선택을 **"Any iOS Device (arm64)"**로 변경
2. 메뉴 **Product > Archive** 클릭 → 빌드 시작 (몇 분 소요)
3. 완료되면 **"Organizer"** 창이 자동으로 뜸
4. **(권장) 업로드 전에 "Validate App"부터 실행**: Organizer에서 방금 만든 아카이브 선택 → **"Validate App"** 클릭 → 배포 방법으로 "App Store Connect" 선택 → 서명 옵션까지 진행하면, 실제 업로드 전에 애플 서버가 미리 검증만 해줘서 흔한 실수(아이콘 누락, Info.plist 오류 등)를 미리 잡아줘요.
5. 문제가 없으면 **"Distribute App"** 클릭 → **"App Store Connect"** 선택 → 계속 진행 (대부분 기본값 그대로 Next) → **"Upload"** 선택
6. 업로드 완료 후 App Store Connect 사이트에서 **"TestFlight"** 탭에 방금 올린 빌드가 보임 (처리에 10분~1시간 정도 걸릴 수 있음, 완료되면 이메일 옴)
7. TestFlight 앱을 본인 아이폰에 설치 → 초대 링크로 팩인백 설치해서 실제 배포 환경과 동일하게 테스트

> TestFlight는 정식 출시 전 최대 90일간 최대 1만 명의 테스터에게 미리 배포해볼 수 있는 애플 공식 기능이에요. 첫 출시 전엔 본인 + appfle님 지인 몇 명 정도로 충분히 테스트해보는 걸 추천해요. **특히 이 단계에서 구글 로그인이 실제로 되는지 꼭 확인하세요.**

## 11. 앱스토어 심사 제출

1. App Store Connect의 앱 페이지에서 **"버전 출시 준비"** 섹션으로 이동
2. 10번에서 업로드한 빌드를 **"빌드"** 항목에서 선택
3. 위에서 채운 스크린샷/설명/개인정보처리방침/앱 프라이버시 등이 모두 채워졌는지 확인
4. **"심사를 위해 제출"** 클릭
5. **로그인이 필요한 앱**이므로, 심사자가 로그인해볼 수 있도록 **"심사 노트(App Review Notes)"** 란에 테스트용 계정의 이메일/비밀번호를 반드시 남겨주세요. (Firebase Auth에 이메일/비밀번호 방식의 테스트 계정을 하나 미리 만들어두고, 짐/가방 예시 데이터도 미리 채워두면 심사가 더 수월해요)
6. 보통 **24~48시간 내** 심사 결과가 나옵니다 (거절되면 사유가 이메일로 옴 → 대응 후 재제출 가능, 재심사는 더 빠른 경우가 많음)

### 비용

|---|---|---|

| Mac 컴퓨터 | 이미 보유 중 | - |

→ **앱스토어 배포 자체에 드는 추가 비용은 연 $99가 전부**예요.

### 자주 막히는 부분

- **구글 로그인이 앱 안에서 안 됨 ("이 브라우저 또는 앱은 안전하지 않을 수 있습니다" / `disallowed_useragent`)** → 0.5번 문제, 반드시 네이티브 구글 로그인 플러그인으로 교체해야 해요. 웹에서는 되는데 앱에서만 안 되면 거의 100% 이 문제예요.
- **"Bundle Identifier가 이미 사용 중"** → `capacitor.config.ts`와 Xcode 양쪽의 `appId`를 더 고유한 문자열로 변경 (예: 뒤에 숫자 추가). 단, 이미 App Store Connect에 등록한 뒤라면 변경 불가하니 신중히.
- **Xcode 26에서 `pod install` 실패 / CocoaPods 관련 빌드 에러** → 터미널에서 `cd ios/App && pod repo update && pod install` 로 강제로 다시 설치해보세요. 그래도 안 되면 CocoaPods를 최신 버전으로 업데이트: `sudo gem install cocoapods`
- **"앱이 iOS 26 SDK로 빌드되지 않았습니다" 류의 업로드 거부** → Xcode가 진짜 26 버전인지 `xcodebuild -version`으로 확인. 여러 Xcode 버전이 설치되어 있다면 `sudo xcode-select -switch /Applications/Xcode.app`로 26 버전을 기본으로 지정.
- **로그인 화면에서 멈춤 (이메일/비밀번호 로그인 등)** → `capacitor.config.ts`의 `server.url`이 정확한 배포 주소인지, Firebase 콘솔의 "승인된 도메인" 목록에 해당 주소가 등록되어 있는지 확인
- **"신뢰할 수 없는 개발자" 에러** (실기기 테스트 시) → 아이폰 설정 > 일반 > VPN 및 기기 관리에서 신뢰 처리
- **심사 거절 (가이드라인 4.2, 최소 기능성)** → 심사 노트에 앱의 핵심 가치(그룹 짐 체크리스트 공유)를 구체적으로 설명하고, 테스트 계정에 예시 데이터를 채워서 보내주세요. 재발되면 오프라인에서도 일부 열람 가능하게 하거나 네이티브 기능을 추가하는 방향으로 보완 후 재제출
- **심사 거절 (가이드라인 4.8, 제3자 로그인)** → 구글 로그인을 제공하면서 Apple 로그인이 없으면 걸릴 수 있어요. 이 문서 9번 항목 참고
- **스크린샷 크기 거부 ("하나 이상의 스크린샷 크기가 잘못되었습니다")** → 위 9번 표의 픽셀 크기(1320x2868 / 2064x2752)를 1픽셀도 틀리지 않게 정확히 맞춰서 다시 캡처
